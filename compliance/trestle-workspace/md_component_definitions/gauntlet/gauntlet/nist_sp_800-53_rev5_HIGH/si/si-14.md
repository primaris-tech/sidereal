# SI-14 — Non-Persistence

## Control Statement

Implement non-persistent operational components that are initialized to a
known state and terminated when no longer needed, preventing accumulation
of residual state that could be exploited by adversaries. Non-persistence
applies to both the components and the data they handle.

## Gauntlet Implementation

Non-persistence is a foundational design property of the Gauntlet probe
execution model, not a configuration option applied after the fact. Probe
runners are structurally incapable of accumulating state between executions.

### Initialized to a Known State

Every probe runner Job starts from a clean, known state:

- **Immutable base image**: Distroless or scratch base with only the
  compiled Rust binary. No configuration files are baked in — the runner
  starts with no history of previous executions.
- **Read-only root filesystem**: `readOnlyRootFilesystem: true` in the
  security context. The container cannot write to its own filesystem.
  Any attempted write to a non-tmpfs path fails at the kernel level.
- **No environment carryover**: Each Job's environment variables are set
  fresh by the controller at Job creation. There is no shared environment
  between executions.
- **Per-execution HMAC key**: A new HMAC key is derived for each Job
  execution from the root secret. The key from a previous execution is
  not present in the new Job's environment.
- **Fresh ServiceAccount token**: Each Job receives a projected
  ServiceAccount token valid only for the duration of the Job. Expired
  tokens from previous executions cannot be reused.

The result: every probe execution is cryptographically and functionally
independent of all previous executions. There is no state a prior execution
could have left behind that affects the current one.

### Terminated When No Longer Needed — TTL-Based Cleanup

Probe runner Jobs complete their single task (probe execution + result write)
and exit. The Kubernetes TTL controller deletes the Job and its Pod
automatically after `ttlSecondsAfterFinished` (default: 300 seconds).

What is deleted:
- The Job resource
- The Pod resource
- The container (and any in-memory state)
- The per-execution HMAC key Secret (deleted by the controller before or
  at TTL expiry)
- The result ConfigMap (read by the controller and converted to a
  `GauntletProbeResult` CR; the ephemeral ConfigMap is then deleted)

After TTL cleanup, no artifact of the probe execution remains in the cluster
except the `GauntletProbeResult` CR (the intentional, HMAC-verified audit
record) and the SIEM export record. There is no residual probe state that
an attacker could inspect to learn about prior execution behavior.

### No Persistent Storage — Admission Policy Enforcement

An admission enforcement policy prevents any Pod in `gauntlet-system` from
mounting a `PersistentVolumeClaim` as a writable volume. This is enforced
at admission — a probe Job spec that includes a writable PVC mount is
rejected before the Pod is created.

Permitted storage mounts for probe runners:
- **HMAC key Secret** (read-only, tmpfs, mounted as
  `/var/run/secrets/gauntlet/hmac-key`)
- **ServiceAccount token** (read-only, projected, auto-mounted by
  Kubernetes)

No other storage mounts are permitted. The prohibition is structural and
not bypassable by modifying a ConfigMap or environment variable after
deployment.

### Non-Persistence of Sensitive Data

The HMAC key is the most sensitive data handled by a probe runner. Its
non-persistence properties:
- Generated fresh per execution (not reused)
- Injected via Secret volume mount to tmpfs (never written to disk)
- Accessible only to the specific Job it was created for (RBAC-scoped)
- Deleted by the controller at Job completion, before TTL expiry
- In-memory lifetime is bounded to the execution window (seconds)

A probe runner that is compromised during execution cannot extract a
long-lived key — the key expires with the execution. A compromised probe
that exits and is replaced by the TTL mechanism starts fresh with a new
key, with no inheritance from the compromised execution.

### Detection Probe Non-Persistence

The Detection Coverage probe runner operates under the strictest
non-persistence posture:
- No network interface (isolated network namespace)
- No volume mounts (not even the HMAC key Secret is needed — the
  detection probe result is a binary observation, not data it signs)
- Seccomp profile blocks all filesystem-modifying syscalls (`write`,
  `creat`, `open` with `O_WRONLY/O_RDWR`, `truncate`, `unlink`)
- The probe runner's only action is emitting a known syscall pattern
  and exiting; all observation is performed by the controller independently

### Enhancement: SI-14(1) — Refresh from Trusted Sources

Every probe execution Job is launched from the cosign-verified, digest-pinned
image (CM-14). The image is pulled from the registry on each Job creation
(or from the node's image cache, where the digest is re-verified against
the admission enforcement policy). The probe runner is always refreshed from
the trusted, signed source — it cannot accumulate drift from a prior
execution's modified state because there is no writable state to modify.

## Evidence Produced

- Kubernetes Job and Pod lifecycle events (creation, completion, TTL deletion)
  in the Kubernetes audit log, exported to SIEM
- Admission controller records confirming that no probe Pod was admitted with
  a writable PVC mount
- Per-execution HMAC key Secret creation and deletion events in the
  Kubernetes audit log
- TTL completion timestamps per Job in the audit log (confirms cleanup
  occurred within the configured window)

## Customer Responsibility

The deploying agency must:
1. Not modify `ttlSecondsAfterFinished` to a very large value or zero
   (which disables TTL cleanup) in their site Helm values override, as
   this would undermine the non-persistence posture and allow probe Pod
   state to accumulate indefinitely
2. Not create exceptions to the admission enforcement policy that prohibits writable PVC
   mounts in `gauntlet-system` without AO authorization
3. Monitor Job TTL cleanup via the Kubernetes audit log to verify that
   probe Jobs are being cleaned up within the configured window
