# CM-14 — Signed Components

## Control Statement

Require that only signed and authenticated components are used within the
system. The signing mechanism must provide integrity assurance and support
non-repudiation. Signature verification must occur at or before deployment.

## Gauntlet Implementation

CM-14 is implemented through a two-part control: cosign signs images at
build time (non-repudiation of provenance), and the admission enforcement
layer (e.g., Kyverno or OPA/Gatekeeper) verifies signatures at admission
time (enforcement before execution). Neither half is optional — a valid
signature exists but is not verified at admission provides no meaningful
protection.

### Part 1: Image Signing at Build Time

Every Gauntlet container image is signed using `cosign sign` during the CI
release pipeline. Key properties of this implementation:

**Signature binding**: Signatures are bound to the image's SHA-256 digest,
not a mutable tag. The signature covers exactly the binary that will be
deployed — a tag mutation cannot produce a valid signature for a different
image content.

**Key management**: The cosign signing key is managed per SC-12 requirements:
stored in the CI pipeline's KMS-backed key store for production releases,
with documented rotation schedule and destruction procedures. The signing
key never exists in plaintext in the CI environment.

**Signature storage**: Signatures are stored as OCI artifacts co-located
with the signed image in the container registry, in the same repository at
`<image-digest>.sig`. This makes them queryable and portable alongside the
image — no separate signature distribution channel is required.

**SBOM attestation**: In addition to the signature, each image receives a
CycloneDX SBOM attestation via `cosign attest --type cyclonedx`. This
creates a chain of custody: the SBOM describes what is in the image, and
the attestation is signed with the same key, providing a single trust root
for both integrity and inventory claims.

### Part 2: Admission Enforcement via Admission Controller

An admission enforcement policy (`gauntlet-image-signature-required` —
rendered for the configured admission controller per the deployment profile)
is deployed as part of the Gauntlet Helm chart. This policy:

1. **Applies to** all Pods in the `gauntlet-system` namespace and all Pods
   anywhere in the cluster that reference a Gauntlet image (by registry
   prefix)
2. **Requires** a valid cosign signature from the Gauntlet signing public key
   before the Pod is admitted
3. **Optionally requires** (configurable) a valid CycloneDX SBOM attestation
   in addition to the signature
4. **Blocks** unsigned or invalidly-signed images at admission — they never
   run, even if somehow pushed to the registry
5. **Generates** an admission policy event (allow/deny) for every
   evaluation, providing a complete record of what was verified and when

The verification key (public) is shipped with the Helm chart and embedded
in the admission enforcement policy. Agencies do not need a separate key distribution
mechanism — the trust root is delivered as part of the Gauntlet deployment
package.

### Independence from the Controller

Critically, the admission controller's image signature verification is
independent of the Gauntlet controller. A compromised controller cannot
bypass or disable the admission enforcement policy — the admission
controller operates as a separate admission webhook (Kyverno evaluates
ClusterPolicies; OPA/Gatekeeper evaluates Constraints). This means image
integrity verification survives a controller compromise.

### Air-Gapped Deployment

For air-gapped environments where Gauntlet images must be mirrored to a
private registry:

1. The agency pulls images from the upstream registry (with signatures)
2. Re-signs mirrored images with a registry-specific key using
   `cosign copy` (which preserves the original signature) or by
   re-signing with an agency-managed key
3. Updates the admission enforcement policy to reference the agency's signing public key
4. The updated policy is documented as a deviation in the SSP

This procedure is documented in the Gauntlet air-gapped deployment guide.

### Non-Repudiation Properties

The cosign signing workflow satisfies non-repudiation requirements:
- The signing key is unique to the Gauntlet release pipeline and not shared
  with any other project or principal
- The Sigstore transparency log (Rekor) records every signing event with
  a timestamped, append-only log entry, providing external proof that an
  image was signed at a specific time
- The signing principal's identity is bound to the log entry
- Log entries cannot be deleted or modified

This means the signing event is externally verifiable and cannot be
repudiated by either the signer or by Gauntlet as an organization.

## Evidence Produced

- cosign signature OCI artifacts for each Gauntlet image, verifiable with
  `cosign verify --key <pub.pem> <image-digest>`
- Sigstore Rekor transparency log entries for each signing event (external,
  append-only, timestamped)
- CycloneDX SBOM cosign attestations for each image
- Admission enforcement policy events (allow/deny) for all Gauntlet image
  deployments in the `gauntlet-system` namespace
- CI pipeline signing step logs per release

## Customer Responsibility

The deploying agency must:
1. Import the Gauntlet signing public key into the admission enforcement
   policy configuration (e.g., Kyverno ClusterPolicy or OPA Constraint)
   using the key distributed with the Helm chart
2. Not disable or create exceptions to the `gauntlet-image-signature-required`
   ClusterPolicy without AO authorization and documentation of compensating
   controls
3. For air-gapped deployments: follow the documented mirroring and re-signing
   procedure and document the agency-specific key as a deviation in the SSP
4. Verify the cosign signature of each Gauntlet release before deployment
   as part of the agency's change management process
