# SC-12 — Cryptographic Key Management

## Control Statement

Establish and manage cryptographic keys for required cryptography in
accordance with applicable laws, directives, policies, regulations,
standards, and guidelines governing key generation, distribution, storage,
access, retirement, and destruction.

## Gauntlet Implementation

Gauntlet uses three distinct categories of cryptographic key material, each
with its own lifecycle requirements. All categories use FIPS 140-2 validated
key management mechanisms.

### Key Category 1: HMAC Probe Result Signing Keys

**Purpose**: Signs probe result payloads before writing to result ConfigMaps;
verified by the controller before accepting any result (SI-7, SC-8).

**Generation**: A unique HMAC key is derived for each probe execution using
FIPS-approved HKDF (HMAC-SHA256) from a root secret stored as a Kubernetes
Secret in `gauntlet-system`. The root secret is generated at Helm install
time using a FIPS-approved CSPRNG. Per-execution keys are ephemeral and
never reused across executions.

**Storage**: The root HMAC secret is stored as a Kubernetes Secret. For
IL4/IL5 deployments, the Helm values include `hmac.kmsKeyArn` (or
equivalent for Azure Key Vault / HashiCorp Vault) to enable envelope
encryption: the root secret in the cluster is encrypted with a KMS data
key; the plaintext key material never exists outside the KMS HSM boundary.

**Distribution**: The per-execution derived key is injected into the probe
runner Job via a Kubernetes Secret mount (read-only, mounted as tmpfs). The
secret is deleted when the Job completes its TTL cleanup. The probe runner
holds the key for the duration of a single execution window.

**Access control**: Only the controller's ServiceAccount can read the root
HMAC Secret. Probe runner ServiceAccounts cannot. The per-execution derived
key Secret is accessible only by the specific Job it was created for.

**Rotation**: The root HMAC Secret is rotated on a configurable schedule
(recommended: 90 days). Rotation is performed via Helm upgrade, which
generates a new root secret and updates the cluster Secret. Old root secrets
are deleted after rotation, as in-flight executions derive their keys from
the new root on the next scheduled execution.

**Retirement/Destruction**: Root HMAC Secrets are deleted from the cluster
on rotation and from KMS on key decommission. KMS key deletion has a
minimum 7-day waiting period (AWS) during which the key can be recovered;
after deletion, key material is permanently destroyed in the HSM.

### Key Category 2: mTLS Client Certificates

**Purpose**: Authenticates the controller to external services (e.g., detection backends
such as Falco or Tetragon, CNI observability endpoints such as Hubble or Calico) and to SIEM endpoints that require mTLS.

**Generation and distribution (SPIFFE/SPIRE preferred)**: For IL4/IL5
deployments, SPIFFE/SPIRE is the recommended certificate authority. SPIRE
issues X.509 SVIDs (SPIFFE Verifiable Identity Documents) to the controller
with a configurable TTL (recommended: 1 hour). Short-lived SVIDs eliminate
the exposure window of a compromised credential — the certificate expires
before an attacker can exploit it.

**Generation and distribution (cluster CA alternative)**: Where SPIRE is
not deployed, client certificates are issued from the cluster's internal CA
via the Kubernetes CertificateSigningRequest API. These have a configurable
validity period (recommended: 90 days). The controller monitors expiry
and alerts via `GauntletSystemAlert` if a certificate approaches expiry
without renewal.

**Rotation**: SVID rotation is continuous and automatic (SPIRE agents
re-key before expiry). Cluster CA certificates are rotated on the configured
schedule. Rotation events are logged to the SIEM.

**Revocation**: SPIRE SVIDs are short-lived enough that revocation is
handled by expiry. Cluster CA certificates are revoked via the Kubernetes
CertificateSigningRequest revocation API. SIEM export of rotation and
revocation events enables audit trail continuity.

### Key Category 3: cosign Image Signing Keys

**Purpose**: Signs Gauntlet container images at release time; verified by
the admission controller at admission (CM-14, SI-7).

**Generation**: The cosign signing key is generated in and managed by a
cloud KMS (AWS KMS, Google Cloud KMS, or Azure Key Vault) backed by a
FIPS 140-2 Level 3 HSM. Key generation occurs in the HSM; the private key
material never leaves the HSM boundary.

**Access**: The signing operation is performed by the CI/CD pipeline's
service identity, which has KMS sign permission. Only the release pipeline
holds this permission. Developers do not have direct signing access.

**Rotation**: Key rotation requires:
1. Generate a new KMS key version
2. Re-sign all active release artifacts with the new key
3. Update the Gauntlet Helm chart `values.yaml` and admission enforcement policy with
   the new public key
4. Publish a signed release note documenting the rotation event
5. Retain the old public key in the admission enforcement policy's `authorities` list for
   verification of previously deployed releases
6. Disable the old key for new signing operations in KMS

**Transparency log**: Every cosign signing event is recorded in the Sigstore
Rekor transparency log — an external, append-only, tamper-evident log.
Rotation events are therefore independently auditable by any party.

**Destruction**: Old KMS key versions are scheduled for deletion after the
rotation retention window (minimum 90 days after all previously signed
images are no longer in active deployment).

### Enhancement: SC-12(1) — Availability

Key availability is maintained through:
- KMS provides HA for HMAC root secret encryption (multi-AZ HSM)
- SPIRE replicates SVID issuance across multiple SPIRE server replicas
- Controller startup validates HMAC Secret availability; `GauntletSystemAlert`
  fires if the Secret or KMS key is unreachable, halting probe execution
  until the key is recovered

### Enhancement: SC-12(2) — Symmetric Keys

HMAC root secrets and SIEM export credential Secrets are symmetric keys
managed per the procedures above. Envelope encryption via KMS ensures the
symmetric key material is protected by an HSM-backed asymmetric master key.

### Enhancement: SC-12(3) — Asymmetric Keys

cosign signing keys and mTLS certificates are asymmetric keys managed per
the procedures above. Private key material for cosign keys never leaves the
KMS HSM. SPIFFE SVIDs use ECDSA P-256 or P-384 (FIPS-approved curves).

## Evidence Produced

- KMS key version history (rotation records) for HMAC root secret KMS keys
- SPIRE SVID issuance logs for mTLS certificates, exported to SIEM
- Controller startup logs confirming HMAC Secret reachability
- `GauntletSystemAlert` CRs for key availability failures
- Sigstore Rekor transparency log entries for each cosign signing event
- Helm chart `values.yaml` key reference history in version control

## Customer Responsibility

The deploying agency must:
1. Configure envelope encryption for the Gauntlet HMAC Secret using their
   KMS (`hmac.kmsKeyArn` in Helm values)
2. Define a key rotation schedule for the HMAC root secret and mTLS
   certificates in their key management policy (recommended: 90 days)
3. Ensure that key destruction procedures for decommissioned keys are
   followed in accordance with their data destruction policy
4. For SPIFFE/SPIRE deployments: operate and maintain the SPIRE server and
   configure SVID TTLs in accordance with their key management policy
5. Document the key management procedures for all three key categories in
   their SSP SC-12 statement
