# SC-13 — Cryptographic Protection

## Control Statement

Determine the cryptographic uses and the type of cryptography required for
each use in accordance with applicable laws, directives, policies,
regulations, and standards. Implement only NIST-validated or NSA-approved
cryptography.

## Sidereal Implementation

FIPS 140-2 validated cryptography is used for every cryptographic operation
in Sidereal. There are no non-FIPS fallback paths. The FIPS posture is
enforced at build time, not configurable at runtime, and verifiable from
the container image SBOM.

### Go Controller — BoringCrypto

The Go controller is compiled with `GOEXPERIMENT=boringcrypto`. This build
tag replaces the standard Go crypto library with BoringCrypto, the
FIPS 140-2 validated module used in Google's production Go toolchain.

BoringCrypto covers all cryptographic operations performed by the controller:

| Operation | Algorithm | Standard |
|---|---|---|
| TLS 1.2/1.3 (external connections) | AES-128-GCM, AES-256-GCM | FIPS 140-2 |
| HMAC result verification | HMAC-SHA256 | FIPS 140-2 |
| HKDF key derivation | HMAC-SHA256-based HKDF | SP 800-56C |
| Digital signature verification (JWT) | RSA-2048+, ECDSA P-256/P-384 | FIPS 140-2 |
| Cryptographic random generation | FIPS-approved DRBG | SP 800-90A |

Non-FIPS algorithms (e.g., RC4, MD5, SHA-1 for signatures, 3DES, AES-CBC
without integrity) are unavailable when BoringCrypto is active. The Go
runtime panics at startup if an attempt is made to use a non-FIPS algorithm
path, preventing silent FIPS mode bypass.

**Verification**: `go tool nm <binary> | grep -i boring` confirms the
BoringCrypto symbols are present. The CI pipeline runs this check as a
build validation step for every release.

### Rust Probe Runners — aws-lc-rs with FIPS Feature

All Rust probe runner binaries are compiled with the `aws-lc-rs` crate's
`fips` feature flag enabled. This links the binary against AWS-LC (Amazon's
fork of BoringSSL), which is FIPS 140-2 validated under CMVP certificate
#4816.

aws-lc-rs covers all cryptographic operations performed by probe runners:

| Operation | Algorithm | Standard |
|---|---|---|
| TLS 1.2/1.3 (to Kubernetes API) | AES-128-GCM, AES-256-GCM | FIPS 140-2 |
| HMAC result signing | HMAC-SHA256 | FIPS 140-2 |
| Random number generation | FIPS-approved DRBG | SP 800-90A |

The Rust build pipeline validates that `aws-lc-sys` links the FIPS-validated
shared library before the build completes. A build that does not link the
FIPS module fails the CI step — it cannot produce a deployable artifact.

**FIPS self-test**: aws-lc-rs runs its FIPS Known Answer Tests (KATs) at
library load time (process startup). A KAT failure causes the process to
exit immediately with a non-zero code, which the Kubernetes Job controller
observes as a failed container. The probe execution never completes, and
the absence of a result is treated as `Indeterminate`, triggering a
`SiderealSystemAlert`.

### S3 Export — SSE-KMS with FIPS-Validated Key

When S3 is configured as a SIEM export target, the controller writes objects
with Server-Side Encryption using AWS KMS (SSE-KMS) and a customer-managed
KMS key. The KMS key must be backed by a FIPS 140-2 Level 3 HSM (AWS KMS
keys backed by CloudHSM satisfy this requirement).

The `siem.s3.kmsKeyArn` field in the Helm values is required when S3 export
is enabled. The controller validates at startup that the KMS key ARN is
present and the key is in an enabled state. A missing or disabled KMS key
prevents S3 export from starting, which triggers a `SiderealSystemAlert`
rather than silently exporting unencrypted.

### FIPS Mode as a Controlled Configuration Setting

FIPS mode is declared as `fips.enabled: true` in the Helm values. This flag
controls which set of container images is deployed — Sidereal ships two
image variants per component:
- `sidereal-controller:v1.x.x-fips` (BoringCrypto-compiled)
- `sidereal-controller:v1.x.x` (standard Go crypto, for non-federal use)

The FIPS variant and the non-FIPS variant are distinct image digests with
distinct cosign signatures. A deployment cannot silently switch between them
without a Helm upgrade. This makes FIPS posture an auditable, change-controlled
setting (CM-2, CM-3, CM-6) rather than a runtime toggle.

### No Non-FIPS Fallback

Sidereal has no code paths that fall back to non-FIPS cryptography on error.
If BoringCrypto or aws-lc-rs fails a self-test or encounters an unsupported
algorithm request, the process terminates rather than falling back. This
design choice is intentional: a failed FIPS module is a critical security
event, not a recoverable error.

### Enhancement: SC-13(1) — FIPS-Validated Cryptography

All Sidereal cryptographic operations use FIPS 140-2 validated modules:
- BoringCrypto: CMVP certificate #3678 (Go 1.19+ production build)
- AWS-LC (aws-lc-rs FIPS): CMVP certificate #4816

CMVP certificate numbers are documented in the release notes for each
Sidereal version and updated when the underlying validated module changes.

## Evidence Produced

- Container build provenance records confirming BoringCrypto and aws-lc-rs
  FIPS compilation flags (CI pipeline logs per release)
- `go tool nm` output confirming BoringCrypto symbol presence (CI validation
  step, archived per release)
- aws-lc-rs FIPS KAT startup log entries in probe runner container logs,
  exported to SIEM
- Controller startup logs confirming FIPS module initialization
- `SiderealSystemAlert` CRs for FIPS module initialization failures
- SBOM listing aws-lc-rs with FIPS feature flag (cosign-attested)

## Customer Responsibility

The deploying agency must:
1. Set `fips.enabled: true` in the Sidereal Helm values and verify that the
   FIPS image variant is deployed (confirmed by image digest in the SBOM)
2. Ensure the KMS key used for S3 SSE-KMS is backed by a FIPS 140-2
   validated HSM (`siem.s3.kmsKeyArn` in Helm values)
3. Confirm that all third-party services Sidereal connects to (e.g., Splunk,
   Elasticsearch, and deployed detection backends such as Falco or Tetragon) are configured to use FIPS-approved cipher
   suites on their server side
4. Document the CMVP certificate numbers for the BoringCrypto and AWS-LC
   modules in their SSP SC-13 statement, referencing the specific Sidereal
   release in use
