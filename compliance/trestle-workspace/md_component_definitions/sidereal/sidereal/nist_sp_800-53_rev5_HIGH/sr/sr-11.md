# SR-11 — Component Authenticity

## Control Statement

Employ tools and techniques to detect counterfeit or illegitimate components
and take specific actions when counterfeit or illegitimate components are
detected. Authenticity verification must occur before component installation
and must be repeatable at any point during the component lifecycle.

## Sidereal Implementation

Component authenticity verification is enforced by two independent mechanisms:
digest pinning in the Helm chart (prevents tag-based substitution at pull
time) and cosign signature verification by the admission controller (e.g.,
Kyverno or OPA/Gatekeeper) (prevents execution of any non-authentic image).
Both mechanisms must pass; neither alone is sufficient.

### Mechanism 1: SHA-256 Digest Pinning

All Sidereal Helm chart image references use the format
`<registry>/<image>@sha256:<digest>`. The digest is the SHA-256 hash of
the image manifest — a cryptographic identity for the exact binary content.

This eliminates tag-based substitution attacks. A counterfeit image pushed
to the same tag in the registry has a different digest. Kubernetes will
attempt to pull `sha256:<original-digest>`, which does not exist for the
counterfeit. The pull fails before any signature verification is attempted.
Digest pinning is the first authenticity gate.

### Mechanism 2: cosign Signature Verification at Admission

Even with digest pinning, an attacker who can write to the registry at the
exact digest could potentially substitute a counterfeit. Cosign signature
verification addresses this: the signature is bound to the digest and signed
with a KMS-backed key that only the Sidereal CI pipeline can access.

The admission enforcement policy (`sidereal-image-signature-required` —
rendered for the configured admission controller per the deployment profile)
enforces signature verification before any Pod in `sidereal-system` is
admitted:

```
Detected as counterfeit/illegitimate:
- Locally built image (no cosign signature)
- Image from an unofficial fork (signed by a different key)
- Registry-mirrored image without re-signing (no signature)
- Image with a valid tag but different digest (signature mismatch)
- Image with a forged signature (cryptographic verification failure)

Action: Admission denied; admission policy event generated; Pod never runs
```

The admission enforcement policy is a Helm pre-install hook requirement —
it is deployed before any Sidereal component Pod is created. There is no
installation window during which an unauthenticated image can run.

### Continuous Authenticity — Not Point-in-Time

A key property of this implementation: authenticity is not verified once
at install time and trusted thereafter. The admission controller verifies
the cosign signature at every Pod admission event — initial deployment,
restart, rescheduling after a node failure, and upgrade.

This means:
- An image that was legitimate at install time is still verified on every
  restart
- A key rotation (SC-12) that invalidates old signatures is immediately
  enforced on the next Pod admission
- An attacker who achieves runtime code execution cannot survive a Pod
  restart without a signed image

### Immutable Registry Path

Sidereal images are published to an OCI registry path where image digests
cannot be overwritten. Once a digest is published and signed, its content
and signature are permanent. An attempt to overwrite the digest would require
writing a new manifest with a different digest — which would not match the
pinned reference in the Helm chart and would not have a valid cosign signature.

This immutability guarantee means the provenance chain from CI build to
deployed binary is preserved in the registry for the lifetime of the release.

### Response to Counterfeit Detection

When the admission controller denies a Pod due to failed authenticity
verification:
1. The Pod is never created — no code executes
2. An admission policy event is generated with the image reference and
   verification failure reason
3. The event is recorded in the Kubernetes audit log
4. Alertmanager rules (configured by the agency) fire on admission policy
   violation events for `sidereal-system` Pods

Agencies must treat an admission controller denial for a Sidereal image as a
**potential supply chain incident**: an unrecognized image attempting to
run as a Sidereal component is a high-severity event requiring immediate
investigation of how that image reference entered the deployment pipeline.

### Enhancement: SR-11(2) — Configuration Control for Component Service

Sidereal images are referenced by immutable digest in the Helm chart (CM-2).
Updates to image references require a Helm upgrade (CM-3), which flows
through the agency's change control process. An unauthorized image reference
cannot be introduced without a traceable change management event.

### Enhancement: SR-11(3) — Anti-Counterfeit Scanning

The Grype/Trivy vulnerability scans (SR-3) run against the SBOM for each
deployed image version. This provides ongoing scanning of the component
inventory for known compromised or counterfeit components (CVEs, malware
signatures in known-bad packages). Scan results are archived per release
and available for assessor review.

## Evidence Produced

- Admission policy records for every Sidereal Pod deployment
  showing cosign verification outcome (allow/deny) with image digest
- cosign signature OCI artifacts for each image, queryable with
  `cosign verify --key <sidereal-pub.pem> <image@sha256:digest>`
- Admission policy violation events for any rejected image admission attempts,
  exported to SIEM
- Helm chart source showing digest-pinned image references per release
  (git history = change record for image references)

## Customer Responsibility

The deploying agency must:
1. Ensure the admission controller (e.g., Kyverno or OPA/Gatekeeper) is
   running and the `sidereal-image-signature-required` policy is active
   before installing Sidereal; verify the policy is in Enforce mode (not
   Audit mode)
2. Store the Sidereal signing public key in a trusted, agency-controlled
   location independent of the Sidereal deployment itself
3. Treat any admission controller denial for a Sidereal image as a potential
   supply chain incident: investigate the source of the unrecognized image
   reference, document findings, and notify the ISSO
4. Monitor for admission policy violation events in the SIEM and configure
   Alertmanager to alert on any violation in the `sidereal-system` namespace
