# CA-7 — Continuous Monitoring

## Control Statement

Develop a system-level continuous monitoring strategy and implement continuous monitoring
in accordance with the organizational continuous monitoring strategy that includes
establishment of defined metrics; establishment of defined frequencies for monitoring
and assessment of control effectiveness; ongoing control assessments; ongoing monitoring
of system and organization-defined metrics; correlation and analysis of information
generated by assessments and monitoring; response actions to address results of the
analysis; and reporting the security and privacy status of the system.

## Sidereal Implementation

CA-7 is the core capability Sidereal is designed to satisfy. Every architectural
decision in Sidereal exists in service of continuous monitoring.

### Monitoring Strategy

Sidereal implements continuous monitoring across five security control surfaces, each
mapped to specific NIST 800-53 controls and MITRE ATT&CK for Containers techniques:

| Control Surface | NIST Controls | Monitoring Mechanism |
|---|---|---|
| RBAC enforcement | AC-2, AC-3, AC-6 | Active API operation attempts; 403 verification |
| Network policy enforcement | SC-7, SC-8 | TCP SYN probes; CNI flow verdict via observability API (e.g., Hubble, Calico) |
| Admission control | CM-6, CM-7 | Known-bad spec submission; webhook rejection verification |
| Secret access boundaries | AC-3, AC-4 | Cross-namespace secret GET attempts; 403 verification |
| Detection coverage | SI-3, SI-4, SI-7 | Adversarial syscall patterns; detection backend (e.g., Falco, Tetragon) alert verification |

### Monitoring Frequencies

Probe execution frequencies are tied to the FIPS 199 impact level declared in Helm
values:

| Impact Level | Maximum Probe Interval |
|---|---|
| High | Every 6 hours |
| Moderate | Every 24 hours |
| Low | Every 72 hours |

The controller enforces compliance warnings if configured cadence exceeds the threshold
for the declared impact level. Probe scheduling includes ±10% jitter to prevent
predictable monitoring windows (SI-4 timing attack mitigation).

### Ongoing Control Assessments

Each probe execution is an automated control assessment. The probe tests whether a
specific security control is **operationally effective** — not merely configured.
This distinction is the fundamental contribution of Sidereal to the continuous
monitoring requirement: existing tools verify configuration; Sidereal verifies
effectiveness.

Every assessment produces a `SiderealProbeResult` with:
- Structured outcome (pass, fail, or indeterminate state)
- Timestamp and probe execution identity
- Specific NIST 800-53 control IDs being assessed
- HMAC-verified integrity status

### Metrics

Sidereal exports the following Prometheus metrics for ongoing monitoring:

| Metric | Description |
|---|---|
| `sidereal_probe_outcome_total` | Count of outcomes by probe type, namespace, and result |
| `sidereal_consecutive_failures` | Consecutive failure count per probe — triggers alerting |
| `sidereal_audit_export_failures` | Failed SIEM exports — indicates retention gap |
| `sidereal_backend_reachability` | Detection/CNI backend availability |
| `sidereal_audit_record_count` | In-cluster audit record count for capacity monitoring |

### Correlation and Analysis

`SiderealProbeResult` records are exported to the configured SIEM with structured
fields enabling correlation by:
- Probe type (control surface)
- Target namespace
- Time window (for trend analysis)
- NIST control family (for control-specific reporting)
- Outcome (for gap identification)

Consecutive failures on a single probe surface across multiple executions surface
the `consecutiveFailures` counter in the `SiderealProbe` status subresource and
trigger Alertmanager rules configured by the deploying agency.

### Response Actions

Control failure outcomes trigger automated response actions:

| Outcome | Automated Response |
|---|---|
| Any failure | SiderealIncident resource created; IR webhook triggered |
| Consecutive failures | Alertmanager rule fires; `consecutiveFailures` counter incremented |
| BackendUnreachable (sustained) | SiderealSystemAlert created; probe execution halted |
| TamperedResult | SiderealSystemAlert created; audit integrity alert fired |

### Reporting

Every `SiderealProbeResult` exported to the SIEM contains the `result.nistControls`
field mapping the probe outcome to specific NIST 800-53 control IDs. This enables
the deploying agency's ISSO to generate continuous monitoring reports directly from
SIEM queries without manual correlation.

OSCAL Assessment Results are generated from `SiderealProbeResult` data, providing
machine-readable evidence directly importable into agency ATO tooling.

### Enhancement: CA-7(1) — Independent Assessment

Sidereal's probe execution is independent of the control implementations it tests.
The controller that verifies detection alerts is separate from the probe runner that
fires them. The controller that records RBAC probe outcomes holds no RBAC privileges
beyond its own operational requirements. This independence ensures assessment results
reflect actual control state.

### Enhancement: CA-7(4) — Risk Monitoring

The `consecutiveFailures` counter and `SiderealIncident` resources provide risk
monitoring inputs. An increasing failure rate on a specific control surface indicates
a degrading security posture that requires risk response before becoming an incident.

## Evidence Produced

- `SiderealProbeResult` records (one per probe execution, SIEM-exported)
- Prometheus metrics (all metrics enumerated above)
- `SiderealIncident` resources (for all control failure outcomes)
- OSCAL Assessment Results (generated from SiderealProbeResult data)
- Probe execution logs in Kubernetes audit log

## Customer Responsibility

The deploying agency must:
1. Declare the system's FIPS 199 impact level in Sidereal Helm values
2. Configure Alertmanager rules for `sidereal_consecutive_failures` and
   `sidereal_audit_export_failures` metrics
3. Configure SIEM queries and dashboards for continuous monitoring reporting
4. Document Sidereal as the continuous monitoring implementation in their SSP CA-7
   statement
5. Conduct the formal organizational continuous monitoring strategy that this
   system-level implementation feeds into
6. Review and respond to SiderealIncident records per their IR procedures
