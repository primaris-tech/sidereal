# AC-12 — Session Termination

## Control Statement

Automatically terminate a session after a defined condition such as an
inactivity period, targeted event, or explicit time limit. Session
termination must apply to all interactive and non-interactive sessions,
including those established by system components and automated processes.

## Sidereal Implementation

Sidereal has no interactive user sessions. All sessions are automated
processes: the controller's persistent API connection and probe runner
Jobs' short-lived API connections. Both are subject to automatic session
termination by design.

### Probe Runner Session Termination — Container Lifecycle

Probe runner sessions are the most straightforward case. Each probe
runner Job runs to completion (one probe execution) and exits. Session
termination is not timeout-based — it is event-based: the session
terminates when the task completes.

The projected ServiceAccount token volume is invalidated when the Job
Pod exits. There is no persistent connection for the session to linger
in after the container terminates. The Kubernetes TTL controller
(SI-14) then deletes the Job and Pod resources, removing all session
artifacts from the cluster.

Maximum session duration for a probe runner is bounded by the probe
execution window (typically 60–120 seconds for most probe types). The
session cannot persist beyond the Job's TTL cleanup window
(`ttlSecondsAfterFinished`, default: 300 seconds after completion).

### ServiceAccount Token Expiry — 1-Hour Hard Limit

All Sidereal ServiceAccount tokens are issued as Kubernetes bound
projected service account tokens. Key properties:

- `expirationSeconds: 3600` (1 hour maximum) — after expiry, the token
  is rejected by the Kubernetes API server
- Audience-restricted: the token is valid only for the intended API
  audience (Kubernetes API server); it cannot be presented to a
  different service for authentication
- Bound to the Pod: the token is cryptographically bound to the specific
  Pod that requested it; a token from a terminated Pod cannot be reused
  in a new context

A token that is compromised (exfiltrated from a running probe runner) has
a maximum useful lifetime of 1 hour from issuance. After expiry, the
Kubernetes API server rejects it unconditionally. The attacker's window
of exploitation is bounded by the token lifetime.

This applies to both controller tokens and probe runner tokens. Long-lived
static tokens for `sidereal-system` ServiceAccounts are not created and
are explicitly prohibited by the agency's AC-12 implementation
requirement below.

### External Session Termination — SPIFFE/SPIRE mTLS

For IL4/IL5 deployments using SPIFFE/SPIRE workload identity, all mTLS
connections to external services (detection backends, CNI observability endpoints, SIEM
endpoints) use X.509 SVIDs with a configurable TTL. Recommended TTL: 1
hour or less. SPIRE enforces automatic credential rotation before SVID
expiry — the SPIRE agent rotates credentials without any application-layer
action required.

Each new mTLS connection from the controller to an external service is
authenticated using the current SVID. When the SVID expires:
- Existing connections using the expired SVID are torn down at the TLS
  layer on the next handshake
- New connections use the freshly rotated SVID
- The previous SVID cannot be reused

This provides continuous session termination for external service
connections: no single session lasts longer than the SVID TTL.

### Secret Access Probe — Expired Token Verification

The Secret Access probe validates that expired or unauthorized tokens
are rejected by the Kubernetes API server. On each execution, the probe
attempts an API operation using a test token crafted to simulate an
expired or cross-audience token presentation. The expected outcome is
a 401 Unauthorized rejection. A `Fail` (success on an expired-token
request) would indicate a session termination enforcement failure in the
API server.

This provides continuous verification that the session termination
mechanism is operating correctly — not just that Sidereal's own tokens
expire, but that the enforcement layer rejects expired tokens.

### Controller Session Management

The controller maintains a persistent connection to the Kubernetes API
server via the controller-runtime watch mechanism. This long-lived
connection is authenticated using a bound ServiceAccount token that
rotates automatically (the token is projected with an expiry; Kubernetes
rotates it before expiry and injects the new token into the running
Pod). The controller does not need to restart to use the rotated token —
the projected volume is updated in-place by the kubelet.

If the token rotation fails (e.g., the kubelet cannot reach the API
server to obtain a new token), the controller's existing connection
eventually expires. The controller then fails health checks, is restarted
by Kubernetes, and re-establishes a fresh authenticated session — automatic
session termination followed by re-initialization.

## Evidence Produced

- Kubernetes audit log entries recording ServiceAccount token issuance,
  renewal, and expiry events, forwarded to SIEM
- `SiderealProbeResult` CRs from the Secret Access probe confirming that
  expired and unauthorized tokens are rejected by the API server
- Job Pod lifecycle events (creation and termination timestamps)
  confirming probe runner session duration in the audit log
- SPIRE SVID issuance and rotation logs for external session credentials
  (when SPIRE is deployed)

## Customer Responsibility

The deploying agency must:
1. Configure the Kubernetes API server with
   `--service-account-max-token-expiration=3600` (or lower) to enforce
   the 1-hour token expiry limit cluster-wide
2. Not create long-lived static tokens (using `kubectl create token
   --duration=87600h` or legacy token Secrets) for any
   `sidereal-system` ServiceAccount
3. For SPIFFE/SPIRE deployments: configure SVID TTLs at or below
   3600 seconds for Sidereal workload identities
4. Review Kubernetes audit log entries for unexpectedly long-lived
   sessions originating from `sidereal-system` ServiceAccounts as
   part of the agency's periodic access review process (AC-2)
